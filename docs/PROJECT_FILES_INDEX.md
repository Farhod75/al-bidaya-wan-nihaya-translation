## Описания файлов

### 1. PROJECT_CONSTITUTION.md
**Назначение:** Главный документ проекта — «конституция». Определяет источники текста, принципы перевода, терминологию, этапы работы и требования к аудиокнигам.  
**Когда читать:** В самом начале. Все остальные документы подчинены этому.  
**Обновляется:** Редко — только при фундаментальных изменениях в проекте.

### 2. AGENTS_STRUCTURE.md
**Назначение:** Описывает 13 агентов: 9 языковых (3 языка × переводчик/редактор/терминолог) + 4 общих (Координатор, Верификатор, QA, Аудио). Для каждого агента — роль, промпт, правила, входы/выходы.  
**Когда читать:** При настройке агентов, при добавлении нового языка.  
**Обновляется:** При изменении структуры агентов.

### 3. VERIFICATION_RESOURCES.md
**Назначение:** Каталог ресурсов для верификации перевода — арабские базы, хадисные сборники, коранические ресурсы, академические источники. С уровнем доверия для каждого.  
**Когда читать:** При верификации, при настройке Верификатора и терминологов.  
**Обновляется:** При открытии новых ресурсов.

### 4. QA_STANDARDS.md
**Назначение:** Стандарты контроля качества для каждого этапа перевода — чек-листы, метрики, уровни серьёзности ошибок, автоматизированные проверки, критерии Pass/Fail. Адаптировано из `hadith-verifier/QA_STANDARDS.md`.  
**Когда читать:** QA Агент читает перед каждой проверкой. Переводчики — перед началом работы.  
**Обновляется:** После каждого цикла перевода (новые метрики, уточнённые пороги).

### 5. FIX_PATTERNS.md
**Назначение:** База знаний типичных ошибок. 16+ паттернов по категориям: терминология, языковой дрифт, исламский контент, технические, стилистические. Формат аналогичен `hadith-verifier/fix_patterns.md` (35+ паттернов).  
**Когда читать:** Каждый агент ОБЯЗАН прочитать перед началом задачи.  
**Обновляется:** После каждого исправления — в том же коммите (правило из hadith-verifier AGENTS.md).

### 6. AB_TESTING_FRAMEWORK.md
**Назначение:** Методология сравнения разных подходов к переводу — промпты, модели, температуры, контексты. Golden Dataset, метрики, LLM-as-Judge, процесс принятия решений. Адаптировано из hadith-verifier Scoring Rubric и LLM-as-Judge.  
**Когда читать:** Перед запуском проекта (обязательные тесты) и при смене модели/промпта.  
**Обновляется:** После каждого A/B теста (добавляется запись в AB_TEST_LOG).

### 7. GITHUB_WORKFLOW.md
**Назначение:** Структура репозитория, naming conventions, процесс коммитов, CI/CD pipeline, правила PR и ревью, организация релизов. Адаптировано из hadith-verifier CI/CD и AGENTS.md git-правил.  
**Когда читать:** При первом коммите и при создании PR.  
**Обновляется:** При изменении инфраструктуры.

### 8. PROJECT_FILES_INDEX.md
**Назначение:** Этот файл. Навигация по всем документам проекта.  
**Когда читать:** Когда нужно найти нужный документ.  
**Обновляется:** При создании нового документа.

## Порядок использования

### При запуске проекта (один раз)

```
1. PROJECT_CONSTITUTION.md     — понять цели, источники, принципы
2. AGENTS_STRUCTURE.md         — настроить агентов
3. VERIFICATION_RESOURCES.md   — подключить ресурсы верификации
4. QA_STANDARDS.md             — настроить проверки
5. AB_TESTING_FRAMEWORK.md     — провести обязательные A/B тесты
6. GITHUB_WORKFLOW.md          — создать репозиторий, настроить CI
7. FIX_PATTERNS.md             — ознакомиться с известными паттернами
```

### При работе над каждым томом

```
1. FIX_PATTERNS.md             — прочитать перед началом
2. QA_STANDARDS.md             — чек-лист этапа
3. GITHUB_WORKFLOW.md          — правила коммитов и PR
4. FIX_PATTERNS.md             — обновить после исправлений
```

### При проблеме/ошибке

```
1. FIX_PATTERNS.md             — поиск существующего паттерна
2. QA_STANDARDS.md             — определить уровень серьёзности
3. FIX_PATTERNS.md             — добавить новый паттерн после исправления
```
